## 索引与证据链（Index & Evidence）

### 为什么需要索引
在企业项目中，Controller 往往不会直接引用 Mapper；数据库操作可能集中在 repo 模块，或分散在 Mapper XML。仅沿调用链向下追容易“断链”。但随着模型能力提升，索引更适合作为“解释与排障”的辅助产物，而不是每次都强制输出的主产物。

### 索引图要覆盖的映射
- PO → table：@TableName("t_xxx") 最优；其次从 XML SQL 提取表名。
- Mapper → PO/table：BaseMapper<T> 泛型绑定；XML namespace 绑定。
- XML statement → table(s)：解析 from/join/into/update 片段。
- Class → depends_on：字段注入、构造器注入、方法调用的静态线索。

### 输出（建议格式）
`index.json` 改为可选调试产物，推荐在以下场景才生成：
- 跨模块符号跳转困难
- XML / DDL / 注解证据互相冲突
- 接口很多，且用户明确要求保留排障索引

若生成 `index.json`，推荐包含：
- tables: [{table, sources:[...]}]
- entities: [{class, table, evidence:[...]}]
- mappers: [{class, entity, xml, evidence:[...]}]
- xmlStatements: [{namespace, id, tables:[...]}]
- dependencies: [{from, to, kind, evidence:[...]}]

controller_report.json 推荐包含：
- controller: {...}
- endpoints: [{
    name, method, pathRaw, pathPostman,
    folders: {...},
    response_contract: {
      wrapper_type,
      wrapper_source,
      success_fields,
      error_fields,
      data_shape,
      confidence,
      evidence
    },
    tables_required: [...],
    tables_possible: [...],
    evidence: [...],
    cases: [...],
    covered_branches: [...],
    uncovered_branches: [...],
    coverage_notes: [...]
  }]
