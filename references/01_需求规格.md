## 需求规格（对齐版）

本 skill 面向“后端无前端页面自测困难”的场景：基于 Controller 自动生成 Postman 用例与 best-effort 的 seed.sql。

### 关键点
- 不依赖严格 DDD。通过项目结构探索与“全局索引图”推导表依赖。
- Postman Collection：鉴权只在顶层定义（bearer {{token}}），子级继承。
- `postman.collection` 为必填，表示目标 Postman Collection 名。当前 Controller 生成结果要作为一个独立 folder 挂到该 Collection 下，而不是单独新建一个 collection。
- `autotest.enabled=true` 时，允许自动执行 `seed.sql -> Postman CLI -> report -> cleanup.sql(按策略)`；自动执行只允许碰 `seed.sql` 与 `cleanup.sql`。
- URL 前缀：来自配置文件 `postman.url_prefix_var`，请求形如 `{{api-base-url}}/v1/...`；`postman.url_prefix_value` 只是可选的兜底默认值，不应用它覆盖 Postman 中现有的同名变量值。
- Path 参数：{id} → :id，并补齐 url.variable，避免导入后请求行为空。
- 输出两份 JSON：导入文件与 API payload。两份 JSON 中的每个 request 都应带只读断言脚本。
- `analysis/index.json` 改为可选调试产物；默认只要求 `analysis/controller_report.json` 提供证据链和覆盖摘要。

### 默认输出目录
- 默认输出到 `.api-test/<ControllerName>_<YYYYMMDD_HHMMSS>/`。

### 合成数据 ID 策略
- 默认使用负数 ID（例如 -9000001 起），避免与线上/测试环境的自增 ID 冲突。
